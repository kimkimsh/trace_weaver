# TraceWeaver — 프로젝트 지침

이 파일은 **이 저장소(trace_weaver)에서만 적용되는** 규칙을 담는다.
일반 코딩 스타일·네이밍·C++ 메모리 정책 등은 사용자 글로벌
`~/.claude/CLAUDE.md`를 그대로 따른다 — 여기서는 중복하지 않는다.

---

## 1. Work Log — 작업 종료 시 반드시 기록

세션에서 코드/문서/설정 등 **무엇이든 변경이 발생했고 그 작업을 마쳤다고
선언하기 직전에**, 다음 규약대로 작업 로그 파일을 생성한다.
"작업 끝" ≒ "로그 작성 완료". 로그 없이 완료를 선언하지 않는다.

### 1.1 디렉토리·파일 구조 (2단)

```
docs/work_log/<topic-slug>/<YYYY-MM-DD>_<short-action>.md
```

- **상위 폴더 `<topic-slug>`** = 작업이 속한 도메인/모듈 단위.
  - kebab-case, 2~4 단어. 예: `scaffold`, `llm-pipeline`,
    `tauri-shell`, `sqlite-schema`, `cli-runner`, `docs-plan`,
    `ci-setup`, `pair-review`.
  - 폴더가 없으면 그 자리에서 만든다 (`mkdir -p` 후 파일 작성).
  - 같은 도메인 작업은 **반드시 같은 폴더에 누적한다**. 새 폴더를
    남발하지 말 것. 애매하면 기존 폴더 목록을 먼저 `ls` 해서 확인.
- **파일 이름 `<YYYY-MM-DD>_<short-action>.md`**
  - 날짜는 항상 절대일자 (상대일자 금지: "오늘"·"어제" → 변환).
  - `<short-action>`은 kebab-case, 2~5 단어, 그날 그 폴더에서 한 일을
    한 줄로 요약. 예: `init-cargo-workspace`, `add-phi4-loader`,
    `fix-trace-overflow`, `migrate-schema-v2`.
  - 같은 날 같은 폴더에 추가 파일이 필요하면 액션명을 다르게 둔다.
    날짜 충돌 시 `_v2` 같은 임시 접미는 금지 — 액션명을 더 구체화.

### 1.2 파일 내용 — 5개 섹션 고정

순서·제목 그대로. 본문은 짧게, 불릿 위주.

```markdown
# YYYY-MM-DD — <세션 한 줄 요약>

## Context
왜 이 세션이 발생했는가. 사용자 요청, 직전 커밋, 외부 트리거 등.
1~3 문단.

## Files changed
변경된 모든 파일을 불릿으로. 각 줄: `경로 (라인 범위) — 한 줄 설명`.
신규 생성·삭제 파일도 포함. 변경 없으면 "없음"을 명시.

## Why
**비자명한 추론**을 적는 자리. 근본 원인, 이 해법을 고른 이유,
대안을 버린 근거, 보존해야 할 불변식. diff에 안 보이는 사고 과정.

## Verification
변경이 동작함을 확인한 방법 — 빌드/타입체크 출력, 실행 명령,
재현 시나리오, grep 결과 등 **재현 가능한 증거**. "테스트했음, 동작함"
같은 모호한 문장은 금지. 검증 못 한 항목은 그렇다고 명시.

## Follow-ups
미완 항목·연기된 결정·도입한 부채·다음 세션 주의사항.
없으면 "None" 이라고 명시 (빈칸 금지).
```

### 1.3 작성 규율 — 반드시 지킨다

- **절대 과거 로그 파일을 수정·삭제하지 않는다.** 사후에 잘못으로 판명
  나도, 새 로그를 쓰고 그 안에서 이전 파일 경로를 인용한다. 역사
  충실성이 로그의 가치다.
- **diff가 보여주는 내용을 반복하지 않는다.** "X를 1에서 2로 바꿈"은
  무가치. "1초 기본값이 Y 경합을 일으켜서 2로 늘림"이 가치 있는 줄.
- **세션이 끝날 때 그 자리에서 쓴다.** "나중에 몰아쓰기" 금지 —
  추론 과정이 휘발된 뒤에 쓰는 로그는 5섹션 형식만 채울 뿐 의미가
  없다.
- **언어**는 한국어/영어 혼용 가능. 섹션 헤더(`## Context` 등)와 코드
  블록·명령어는 영어로 고정.
- **트리비얼한 변경**도 짧게라도 남긴다. 10줄짜리 로그도 충분.
  "사소해서 안 적었다"가 다음 세션에서 같은 실수를 반복하게 만든다.

### 1.4 세션 시작 시 점검

비자명한 작업을 시작하기 전, 다음을 먼저 확인한다.

```bash
# 최근 작업 로그 (모든 토픽 폴더 통합, 최신 5개)
find docs/work_log -name '*.md' -type f -printf '%T@ %p\n' \
  | sort -rn | head -5 | cut -d' ' -f2-

# 토픽 폴더 일람 (새 폴더 만들기 전 중복 방지)
ls docs/work_log/ 2>/dev/null
```

직전 세션의 `Follow-ups`가 현재 요청과 겹치면 그 로그 경로를 응답에
인용해 사용자에게 컨텍스트 체인을 보여준다.

### 1.5 안티 패턴 (거부)

- "커밋 메시지에 다 적었으니 로그는 생략" → 거부. 커밋 메시지는
  ~70자, 5섹션 추론 구조를 담을 수 없다.
- "오늘 변경 전부 하나의 로그로 일괄 작성" → 거부. 세션 단위 = 토픽
  단위로 분리해서 쓴다.
- "폴더 만들기 귀찮아서 `docs/work_log/` 직속에 파일을 둠" → 거부.
  반드시 토픽 폴더를 거친다 (위 §1.1).
- "지난 로그가 틀려서 지웠음" → 거부. 새 로그로 교정.
- "`docs/work_log/`가 git에 안 올라가니 다른 디렉토리로 옮김" → 거부.
  의도된 위치다. 사용자가 명시적으로 옮기라 지시할 때만 변경.

## 2. 프로젝트 컨텍스트 (요약)

- **목표**: Linux 개발자를 위한 로컬 컨텍스트 인프라 (TraceWeaver).
- **스택 (현 계획)**: Rust 2024 / SQLite / Tauri / Phi-4 mini.
  자세한 사양·로드맵·리스크는 `docs/plan/00_overview.md` 이하 참조.
- **MVP**: 4주 목표. Claude + Codex(GPT-5.5/xhigh) 페어 리뷰로 합의됨
  (`docs/plan/06_pair_review.md`).
- 코드 작성 시 위 계획 문서가 1차 진리이며, 변경 시 해당 문서와
  `docs/work_log/`에 동시 반영한다.

---

## 3. 보조 자료 위치

- `docs/background/` — 사전 조사·배경 문서 (변경 거의 없음).
- `docs/ideation/` — 초기 아이디에이션 산출물 (역사 자료).
- `docs/subject/` — 과제/공모 원문 자료.
- `docs/plan/` — **현행 계획·아키텍처·로드맵 (1차 진리).**
- `docs/work_log/` — 세션별 작업 로그 (이 파일 §1 규약).
