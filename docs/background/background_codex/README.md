## 핵심 배경 요약

AI 앱/서비스의 환경은 "챗봇 기능 추가"에서 "AI가 앱의 실행 계층이 되는 구조"로 이동하고 있다. 소비자 시장에서는 범용 AI 어시스턴트가 기본 진입점이 되고, 모바일에서는 생성형 AI 앱의 다운로드/수익/사용시간이 동시에 커졌다. 기업 시장에서는 AI 도입률은 높지만 대다수 조직이 아직 전사적 가치 실현보다 파일럿/부분 배포 단계에 머문다.

PC와 모바일의 하드웨어 측면에서는 NPU, 온디바이스 모델, 브라우저 내장 AI, 모델 배포 파이프라인이 빠르게 성숙하고 있다. 하지만 온디바이스 AI는 클라우드 모델의 단순 대체재가 아니라 개인정보, 오프라인, 지연시간, 비용을 줄이는 보조 실행 계층에 가깝다. 앱 설계는 클라우드 모델, 온디바이스 모델, 검색/RAG, 외부 도구 호출을 섞는 하이브리드 구조가 기본 전제가 된다.

데이터는 AI 서비스의 경쟁력을 결정하지만, 크롤링은 기술 문제가 아니라 권리, 출처, 개인정보, 품질, 보안까지 포함하는 운영 문제다. `robots.txt`는 접근 허가가 아니며, 외부 웹 문서는 프롬프트 인젝션과 데이터 오염의 공격면이 될 수 있다. 크롤링 단계부터 출처, 라이선스, 수집 시각, 정제 규칙, 재현 가능한 파이프라인을 기록해야 한다.

## 배경조사에서 얻은 CTO 관점의 판단 기준

- AI 기능은 모델 호출보다 데이터 접근권, 워크플로우 통합, 신뢰성 평가, 권한 제어가 더 큰 차이를 만든다.
- 소비자 앱은 "신기함"만으로 유지되지 않는다. 사용 빈도, 마찰 감소, 신뢰, 기본 어시스턴트와의 차별적 맥락이 중요하다.
- 기업/프로슈머 앱은 정확도, 지연시간, 보안, 감사 가능성이 제품성의 일부다.
- 모바일/PC 앱은 클라우드 API 비용보다 배터리, 모델 다운로드, 기기 호환성, 앱스토어 심사 정책이 실제 출시 리스크가 될 수 있다.
- 에이전트형 기능은 편리하지만, 권한과 행동 범위가 넓어질수록 보안 설계가 제품의 핵심 아키텍처가 된다.
- 크롤링한 데이터는 모두 불신 입력으로 처리해야 한다. 저장 전에 정제하고, 검색 전에 품질을 평가하고, 모델에 넣기 전에 출처와 권한을 확인해야 한다.
