# ContextD: 리눅스 개발자 활동 기반 AI 컨텍스트 & 스킬 자동 생성 데몬 (간이 기획서)

> **프로젝트 명 (가칭):** ContextD / OmniTrack Linux / DevSense
> **신규 프로젝트 명 후보:** 
> 1. **TraceWeaver (트레이스위버):** 개발자의 모든 흔적(Trace)을 엮어(Weave) AI의 문맥으로 만들어준다는 의미.
> 2. **Synapse-OS (시냅스-OS):** OS 단에서 돌아가며 뇌의 시냅스처럼 개발자의 터미널, 브라우저, 에디터 활동을 하나로 연결.
> 3. **SilentScribe (사일런트스크라이브):** 리눅스 백그라운드에서 조용히(Silent) 개발자의 문제 해결 과정을 기록(Scribe)하는 서기.
> 4. **ContextFS (컨텍스트FS):** 리눅스의 파일 시스템(ext4 등) 네이밍을 차용하여, '문맥(Context)을 관리하는 시스템'이라는 개발자 친화적 네이밍.
> 5. **Aura-Dev (아우라-데브):** 개발자 주변에 보이지 않게 감도는 분위기(Aura)처럼, 항상 백그라운드에서 상주하며 컨텍스트를 파악.
> **플랫폼:** Linux 데스크톱 / 서버 백그라운드
> **목적:** OASIS 아이디어를 리눅스 개발자 환경으로 피버팅(Pivoting)하여, 개발자의 모든 활동을 수집하고 AI 에이전트용 '스킬'과 '프롬프트'를 자율 생성하는 생산성 도구 기획.

---

## 1. 프로젝트 개요

리눅스 개발 환경에서 일어나는 모든 활동(터미널 명령어, 코드 수정, Git 커밋 내역, 웹 브라우징 및 에러 검색 등)을 시스템 백그라운드에서 실시간으로 수집하고 분석하는 프로그램입니다. 

수집된 데이터를 단순히 보여주는데 그치지 않고, 반복되는 개발 워크플로우를 파악하여 현존하는 AI 에이전트(Claude Code, Cursor, Copilot 등)들이 즉각 사용할 수 있는 **'표준화된 에이전트 스킬(Agent Skills)'**과 **'상황 맞춤형 프롬프트'**로 자동 변환(Auto-generation)해 주는 것이 핵심입니다.

---

## 2. 타겟 고객 및 해결하려는 문제점

### 타겟 고객
*   CLI 중심의 워크플로우를 선호하는 리눅스(Ubuntu 등) 환경의 소프트웨어 엔지니어, DevOps, 데이터 엔지니어.

### 문제점 (Pain Points)
1. **단편화된 컨텍스트와 반복되는 설명:** 에러가 발생했을 때 AI에게 질문하려면, "지금 브랜치 상태가 어떻고, 방금 터미널에서 어떤 명령어를 쳤고, 무슨 에러 로그가 났는지" 사용자가 매번 복사/붙여넣기로 수동 설명해야 합니다.
2. **에이전트 스킬(Skill) 작성의 높은 허들:** 최신 에이전트들은 반복 작업을 `SKILL.md` 같은 형태로 등록해 자동화할 수 있습니다. 그러나 개발자가 자신의 워크플로우를 분석해 이 지침서를 직접 작성하는 것 자체가 귀찮고 시간이 드는 일입니다.
3. **회사 코드 유출 우려 (Shadow AI):** 엔터프라이즈 환경에서는 개발 활동 로그나 소스 코드를 클라우드 AI 서버로 전송하는 것에 대한 극도의 보안 거부감이 존재합니다.

## 3. 핵심 기능 (Core Features)

### 3.1. 전방위 백그라운드 트래커 (Omni-Tracking Daemon)
*   **시스템 레벨 로깅:** Linux 커널 기능(eBPF, inotify 등)을 활용해 파일 시스템의 변경 사항과 터미널(Bash/Zsh) 명령어 실행 히스토리를 오버헤드 없이 수집합니다.
*   **개발 브라우징 수집:** 개발자가 스택오버플로우, GitHub, 공식 문서 등에서 무엇을 검색하고 읽었는지 로컬 익스텐션을 통해 수집합니다.
*   **로컬 보안:** 수집된 모든 데이터는 절대 외부 서버로 전송되지 않으며, 사용자 PC의 로컬 벡터 DB(SQLite/Chroma 등)에 암호화되어 저장됩니다.

### 3.2. 실시간 개발자 대시보드 (Developer Dashboard)
*   오늘 하루 어떤 파일에서 가장 오래 머물렀는지, 어떤 유형의 에러(예: 컴파일 에러 40%, 런타임 에러 60%)를 겪고 해결했는지 타임라인 형태로 시각화하여 제공합니다.
*   자신의 생산성 병목(Bottleneck) 구간을 객관적인 데이터로 확인 가능합니다.

### 3.3. AI 스킬(Skill) & 프롬프트 자동 생성기 (Auto-Skill Creator)
*   **워크플로우 패키징:** 특정 버그 해결이나 배포 환경 세팅을 위해 개발자가 입력한 '웹 검색 10분 → 코드 수정 3번 → 터미널 명령어 5번'의 과정을 로컬 AI(SLM)가 패턴화합니다.
*   **SKILL.md 자동 생성:** 추출된 패턴을 바탕으로, "다음에도 동일한 파이프라인 에러가 발생하면 이 명령어 순서대로 실행해"라는 표준화된 에이전트 지침서(`SKILL.md` 포맷)를 자동 생성하여 로컬 에이전트 디렉토리에 등록합니다.
*   **동적 프롬프트 추천 (Dynamic Prompt Injection):** 지금 막 에러가 터졌을 때 대시보드를 열면, AI가 알아서 *"현재 `src/main.rs`의 42번째 줄 수정 후 `cargo build`에서 난 에러에 대해 질문할까요?"* 라고 완벽한 컨텍스트가 담긴 프롬프트를 미리 완성해 두고 제안합니다.

## 4. 권장 기술 스택 (Technical Architecture)

*   **Daemon & Backend:** `Rust` 또는 `Go` (리눅스 커널 자원 제어에 용이하며 시스템 리소스 점유율을 최소화하기 위함)
*   **Local AI Engine:** `llama.cpp` 또는 `Ollama`를 내장하여, Qwen Coder 3B나 Llama 3.2 3B 같은 가벼운(SLM) 로컬 모델로 백그라운드 분석을 수행 (프라이버시 완벽 보장).
*   **Frontend (Dashboard):** `Tauri` (Rust 기반의 초경량 데스크톱 앱 프레임워크) + `React`/`Svelte`. 기존 Electron 대비 메모리 점유율이 압도적으로 낮아 백그라운드 실행에 적합합니다.

---

## 5. 차별화 및 비즈니스 가치 (Value Proposition)

*   **"Show, Don't Tell":** 기존에는 개발자가 AI에게 자신의 상황을 일일이 말해줘야(Tell) 했다면, **ContextD**는 운영체제 단에서 개발자를 지켜보고(Show) 스스로 컨텍스트를 조립합니다. AI 사용의 가장 큰 허들인 '질문하기 전의 맥락 셋업' 과정을 0초로 단축시킵니다.
*   **AI 도구의 허브(Hub) 역할:** ContextD 자체가 완벽한 AI 비서일 필요는 없습니다. 이 데몬이 생성한 고품질의 컨텍스트와 스킬 파일들을 개발자가 기존에 쓰던 툴(Claude Code, Cursor 등)에 플러그인처럼 밀어 넣어주는 **'AI를 위한 AI 인프라'**로서 기능합니다.
*   **철저한 온디바이스(On-device):** 오직 리눅스 로컬 환경에서 돌아가는 폐쇄망 지향 구조로, 기업과 프리랜서들의 코드 유출 걱정을 원천 차단하는 강력한 B2B 어필 포인트가 됩니다.
